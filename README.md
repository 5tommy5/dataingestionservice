# DataIngestionService

A .NET 8 Web API for ingesting, validating, deduplicating, and querying customer transaction data.

## Quick Start

```bash
# Full stack (API + PostgreSQL + Redis)
docker compose up

# API only
dotnet run --project DataIngestionService.Api

# Tests
dotnet test
```

## Architecture

Clean Architecture with four layers: `Domain` ŌåÆ `Application` ŌåÆ `Infrastructure` / `Api`.

- **Domain** ŌĆö entities and domain exceptions; no external dependencies
- **Application** ŌĆö use cases, DTOs, FluentValidation validators, repository/cache interfaces
- **Infrastructure** ŌĆö EF Core (PostgreSQL), bulk inserts via `EFCore.BulkExtensions`, Redis cache
- **Api** ŌĆö ASP.NET Core 8 controllers, DI wiring

Deduplication uses a SHA-256 `idempotency_key` (`customer_id|datetime|amount|currency|channel`) stored as a UNIQUE constraint ŌĆö duplicates are silently skipped via `ON CONFLICT DO NOTHING`. Batch ingestion streams CSV line-by-line and inserts in chunks of 15,000 rows. Stats are Redis-cached with a 60s TTL.

## Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | `/ingest/batch` | Multipart CSV upload |
| POST | `/ingest/transaction` | Single JSON transaction |
| GET | `/customers/{id}/transactions` | Paginated transaction list |
| GET | `/stats/summary` | Aggregate stats (Redis-cached, 60s TTL) |

See [docs/prd.md](docs/prd.md) and [docs/plan.md](docs/plan.md) for full requirements and implementation plan.

## Trade-offs

- **Stale stats cache** ŌĆö Redis TTL (60s) means stats can lag behind recent ingests. A write-through or invalidation-on-ingest approach would be more accurate but adds complexity.
- **Redis vs in-memory cache** ŌĆö for a single API instance, in-memory cache would be simpler and faster (no network hop, no serialization). Redis only pays off with multiple instances or when cache needs to survive restarts.
- **Duplicate counting in bulk ingest** ŌĆö `BulkInsertOrUpdateAsync` performs a no-op update (no columns updated) for duplicate rows matched by `idempotency_key`. These are counted as rejected via `StatsNumberInserted`, but the response gives no indication of which rows were duplicates or which existing record they conflicted with.
- **Batch ingestion blocks the HTTP request** ŌĆö `POST /ingest/batch` holds the connection open until the entire file is processed: every row is validated with FluentValidation, then bulk-inserted in chunks of 15,000 rows. For large files this can exceed HTTP timeouts (Kestrel default, reverse proxy limits such as nginx or Azure Application Gateway). The correct fix is not to increase timeouts but to decouple ingestion from the HTTP request ŌĆö return `202 Accepted` with a job ID immediately and process the file in a background worker (`IHostedService` + `Channel<T>`).

## What I'd do differently with more time

- Make the bulk insert chunk size and cache TTL configurable via `appsettings.json`.
- Add structured logging and a health-check endpoint (`/healthz`) for production readiness.
- Move the bulk upsert logic into a PostgreSQL stored procedure to get proper per-row duplicate tracking and cleaner separation between application and persistence concerns.
- Async batch processing ŌĆö large CSV uploads currently block the HTTP request until the entire file is processed. A better approach would be a two-request flow: the first call returns `202 Accepted` with a job ID, and a second request polls for the result (accepted/rejected counts, errors). This could be backed by an outbox pattern or a message broker for resilience.
- Move credentials (database connection strings, Redis URLs) out of `appsettings.json` / environment variables and into a secrets vault (e.g. HashiCorp Vault, AWS Secrets Manager, or Azure Key Vault) to avoid credentials being stored in plaintext in config files or leaking through environment inspection.

## AI Usage

AI (Claude) was used in the following ways during this project:

- **PRD drafting** ŌĆö initial product requirements document generated from a brief description of the service goals
- **Implementation plan** ŌĆö high-level plan and task breakdown generated from the PRD
- **Review and corrections** ŌĆö both the PRD and plan were reviewed manually; notable corrections made included: enforcing Clean Architecture layering (no EF Core / Redis leaking into Application/Domain), scoping Redis cache only to the stats summary endpoint, replacing regex-based currency validation with real ISO 4217 data via the `ISOCurrencies` NuGet package, and wiring FluentValidation explicitly in use cases rather than relying on model binding auto-validation
- **Project structure** ŌĆö solution scaffold, project references, and NuGet package selection generated by AI (subtask 1 of the implementation plan)
- **DTOs and interfaces** ŌĆö all Application layer DTOs (`TransactionRequest`, `BatchIngestResponse`, `CustomerTransactionsResponse`, `StatsSummaryResponse`, etc.) and repository/cache interfaces (`ITransactionRepository`, `IStatsCache`) generated by AI (subtask 3 of the implementation plan)
- **Manual review ŌĆö `StatsSummaryResponse`** ŌĆö the initial AI-generated `StatsSummaryResponse` was incomplete (only had `TotalTransactions`, `TotalAmount`, `UniqueCustomers`); reviewed manually against the PRD and corrected: added `TotalVolumeByCurrency`, `TopSourceChannels`, `TransactionsLast24h`, renamed `UniqueCustomers` ŌåÆ `TotalCustomers`, and changed `AvgTransactionAmount` to `AvgDailyTransactionCount` (average total per day over the last 30 days) ŌĆö this change was made by hand and also reflected back into the PRD
- **Idempotency key refactor** ŌĆö AI initially duplicated the SHA-256 idempotency key computation inside both `IngestTransactionUseCase` and `IngestBatchUseCase`; reviewed manually and extracted into a single `IdempotencyKeyHelper.Compute()` static helper. Also noticed that the key used only the date part of `TransactionDate` (`yyyy-MM-dd`), discarding the time ŌĆö since two transactions from the same customer with the same amount, currency, and channel could legitimately occur at different times on the same day, the full timestamp (`yyyy-MM-ddTHH:mm:ss`) is needed for correct deduplication. The fix required changing `transaction_date` from `DATE` to `TIMESTAMP` across the data model, validator, and docs.
- **Infrastructure repository and cache** ŌĆö `TransactionRepository` and `RedisStatsCache` implementations (subtask 8) generated by AI. Manually reviewed and changed `TopSourceChannels` ranking from count-only to total transaction amount (`SUM(amount)`), which better reflects channel value rather than just volume.
- **Bulk insert and batch stats** ŌĆö the bulk insert logic and the statistics returned from batch ingestion were manually changed after AI generation. This includes adjustments to how the bulk operation is executed and how accepted/rejected counts are tracked and reported in the `BatchIngestResponse`.
- **Unit tests** ŌĆö the xUnit test suite was generated by AI, covering use cases, validators, and repository logic. Tests were reviewed manually before being committed.
- **CSV delimiter validation** ŌĆö initial implementation used manual delimiter detection; changed to rely on CsvHelper parsing and validate that the parsed header contains the required 5 columns. This includes the `InvalidCsvFormatException` domain exception, detection in `IngestBatchUseCase`, a `400 Bad Request` response in `IngestController`, and a corresponding unit test.
- **AI code review** ŌĆö the full codebase (both AI-generated and manually written code) was reviewed by AI, which identified several issues that were subsequently fixed.
- **CancellationToken fix** ŌĆö AI added `CancellationToken` support to batch ingestion but only threaded it through to `BulkInsertAsync`. Since CsvHelper's `ReadAsync` does not accept a token, the row loop would keep parsing until the next chunk boundary (up to 15,000 rows), at which point `BulkInsertAsync` would throw `OperationCanceledException`. Manually corrected by adding `cancellationToken.ThrowIfCancellationRequested()` inside the row loop so processing stops on every row, not just at chunk boundaries.
