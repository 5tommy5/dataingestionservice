# DataIngestionService

A .NET 8 Web API for ingesting, validating, deduplicating, and querying customer transaction data.

## Quick Start

```bash
# Full stack (API + PostgreSQL + Redis)
docker compose up

# API only
dotnet run --project DataIngestionService.Api

# Tests
dotnet test
```

## Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | `/ingest/batch` | Multipart CSV upload |
| POST | `/ingest/transaction` | Single JSON transaction |
| GET | `/customers/{id}/transactions` | Paginated transaction list |
| GET | `/stats/summary` | Aggregate stats (Redis-cached, 60s TTL) |

See [docs/prd.md](docs/prd.md) and [docs/plan.md](docs/plan.md) for full requirements and implementation plan.

## AI Usage

AI (Claude) was used in the following ways during this project:

- **PRD drafting** — initial product requirements document generated from a brief description of the service goals
- **Implementation plan** — high-level plan and task breakdown generated from the PRD
- **Review and corrections** — both the PRD and plan were reviewed manually; notable corrections made included: enforcing Clean Architecture layering (no EF Core / Redis leaking into Application/Domain), scoping Redis cache only to the stats summary endpoint, replacing regex-based currency validation with real ISO 4217 data via the `ISOCurrencies` NuGet package, and wiring FluentValidation explicitly in use cases rather than relying on model binding auto-validation
- **Project structure** — solution scaffold, project references, and NuGet package selection generated by AI (subtask 1 of the implementation plan)
- **DTOs and interfaces** — all Application layer DTOs (`TransactionRequest`, `BatchIngestResponse`, `CustomerTransactionsResponse`, `StatsSummaryResponse`, etc.) and repository/cache interfaces (`ITransactionRepository`, `IStatsCache`) generated by AI (subtask 3 of the implementation plan)
- **Manual review — `StatsSummaryResponse`** — the initial AI-generated `StatsSummaryResponse` was incomplete (only had `TotalTransactions`, `TotalAmount`, `UniqueCustomers`); reviewed manually against the PRD and corrected: added `TotalVolumeByCurrency`, `TopSourceChannels`, `TransactionsLast24h`, renamed `UniqueCustomers` → `TotalCustomers`, and changed `AvgTransactionAmount` to `AvgDailyTransactionAmount` (average total per day over the last 30 days) — this change was made by hand and also reflected back into the PRD
- **Idempotency key refactor** — AI initially duplicated the SHA-256 idempotency key computation inside both `IngestTransactionUseCase` and `IngestBatchUseCase`; reviewed manually and extracted into a single `IdempotencyKeyHelper.Compute()` static helper. Also noticed that the key used only the date part of `TransactionDate` (`yyyy-MM-dd`), discarding the time — since two transactions from the same customer with the same amount, currency, and channel could legitimately occur at different times on the same day, the full timestamp (`yyyy-MM-ddTHH:mm:ss`) is needed for correct deduplication. The fix required changing `transaction_date` from `DATE` to `TIMESTAMP` across the data model, validator, and docs.
